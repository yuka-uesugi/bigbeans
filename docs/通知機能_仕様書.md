# 通知機能 仕様書

作成日: 2026-06-09
作成者: クロコA（PM）
対象アプリ: bigbeans（バドミントンサークル運営アプリ）

このファイルは「通知機能」の決定事項と実装方針をまとめた仕様書です。
新しいセッションで通知機能の続きをやるときは、まずこれを読んでください。

---

## 1. 背景・目的

カレンダーに新しい予定が追加されたとき、アンケートが追加されたときなど、
**メンバー全員に必ずお知らせが届く**ようにしたい。

メンバー層はPCが苦手・ご年配が多いため、「設定不要で確実に届く」ことを最優先する。

---

## 2. オーナー（上杉さん）が決めた方針（2026-06-09）

### (A) 通知の出口（チャンネル）— 3つを併用
1. **アプリ内のベル通知**（全員・常時・設定不要）… まず必ず作る土台
2. **メール通知**（確実に届く）… ただし **各メンバーが「受け取る／受け取らない」を自分で設定できる**ようにする
3. **スマホのポップ通知（PWA）**（希望者向け）… ホーム画面に追加した人がポップで受け取れる
   - ※ LINE通知は「手間が大きい」ため今回は採用しない（非推奨）

### (B) 通知を送る「きっかけ」（トリガー）— 4つ
1. カレンダー予定の追加
2. アンケートの追加
3. お知らせ（announcements）の追加
4. 意見箱への返信（※既存。すでに動いている個人向け通知）

### (C) 進め方
**第1段階から順番に作る**（一気にやらず、確実なものから積み上げる）

---

## 3. 段階プラン（実装ロードマップ）

### 第1段階【アプリ内ベル通知を「全員向け」に】… 完了（2026-06-09）
- [x] 予定・アンケート・お知らせが追加されたら、**全員のベル**に自動表示
- [x] 既存の「意見箱への返信」通知（個人向け）も引き続き動く（パス不具合も修正）
- [x] 画面右上のベル（TopBar）を本物にした（未読件数バッジ・相対時刻・タップで該当画面へ）
- [x] サイドバーの未読バッジも本物の件数（個人＋全員向けの合算）
- 費用ゼロ・メンバーの追加設定不要

**実装ファイル（第1段階）**:
- `src/lib/notifications.ts` … broadcasts 発行・購読、最終既読時刻、個人通知のパス統一
- `src/hooks/useNotificationFeed.ts` … 個人＋全員向けを1つに束ねる共通フック・相対時刻
- `src/components/dashboard/TopBar.tsx` … ベルのドロップダウンを本物に
- `src/components/dashboard/Sidebar.tsx` … 未読バッジを共通フックに切替
- トリガー: `AddEventModal.tsx` / `SurveyCreateModal.tsx` / `AnnouncementsBoard.tsx`
- `firestore.rules` … `broadcasts` と `notificationState` のルール追加（**要・コンソール反映**）

**要・反映作業（次に本番反映するとき）**:
- firestore.rules を Firebaseコンソールの「ルール」タブで公開（broadcasts・notificationState を追加済み）。
  これを公開しないと、本番では通知の保存・取得が拒否される。

### 第2段階【メール通知】
- 必須のお知らせ（予定追加・アンケート・お知らせ）をメールでも送る
- **各メンバーがマイページで「メール通知を受け取る／受け取らない」を設定**できる
- 送信には送信サービス（例: Resend / SendGrid の無料枠、または Firebase 拡張）を1つ接続する
- 全員への一斉送信はサーバー側処理が必要（Cloud Functions など）になる見込み

### 第3段階【スマホのポップ通知（PWA）】… 任意・希望者向け
- Service Worker と FCM（Firebase Cloud Messaging）を導入
- ホーム画面に追加した人が、スマホにポップ通知を受け取れる
- 各自で「ホーム画面に追加」＋「通知を許可」の初期設定が必要
- iPhone は iOS 16.4 以降で、ホーム画面に追加したPWAのみ対象（制約あり）

## 4. 第1段階の技術設計

### データの持ち方
- **全員向け通知**は「ブロードキャスト」方式にする。
  - 新コレクション `broadcasts/{id}` に1件＝1お知らせとして保存
  - 各メンバーは `broadcasts` を購読して全員が同じお知らせを見る
  - 「誰が読んだか」は各自の最終既読時刻 `lastReadBroadcastAt` で管理（未読＝それより新しいお知らせ）
  - メンバーのFirebase UID 一覧を持たなくてよい（名簿はメール基準でUID未確定の人もいるため）

- **個人向け通知**（意見箱への返信）は従来どおり各自のサブコレクションに保存。
  - ただし保存先パスを **`users/{uid}/notifications/{id}`** に統一する（後述の不具合修正）。

### broadcasts のデータ構造（案）
```
broadcasts/{id} {
  type: "event" | "survey" | "announcement",
  title: string,        // 例: 新しい予定: 4/12 練習
  body?: string,        // 補足
  link: string,         // タップ時の遷移先（例: /dashboard/calendar）
  createdByName?: string,
  createdAt: Timestamp,
}
```

### トリガーの置き場所（重要）
- ブロードキャスト発行は **各作成モーダルの呼び出し側**（AddEventModal / SurveyCreateModal / お知らせ作成UI）で行う。
  - 理由: 予定の一括登録（seed）など、まとめ追加で全員に大量通知が飛ぶのを避けるため。
  - 手動の1件追加だけ通知する。

### 既存の不具合（第1段階で同時に修正）
- 返信通知のコードは `notifications/{uid}/items` に書き込んでいるが、
  ルールは `users/{uid}/notifications` を許可しており **パスが食い違っている**。
  → 本番で返信通知が拒否されていた可能性。コード側を `users/{uid}/notifications` に合わせて修正する。
  - 影響ファイル: `src/lib/notifications.ts`, `SuggestionBox.tsx`, `Sidebar.tsx`, `profile/page.tsx`

### firestore.rules の追加（要・コンソール反映）
- `broadcasts/{id}`: read=メンバー, create=メンバー, update/delete=admin（または不可）
- `notificationState/{uid}`（または users ドキュメント内のフィールド）: 本人のみ read/write
- ※ ルール反映は Firebaseコンソールの「ルール」タブで行う（CLIは権限なし。storage の時と同じ）

## 5. 反映待ち・注意

- firestore.rules に通知関連ルールを追記したら **コンソールで公開**が必要（未公開だと拒否される）
- 既存の `paymentCollections`・`reports削除` の未公開分とまとめて公開してよい
