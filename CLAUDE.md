## このアプリについて
- 名前: bigbeans（ビックビーンズ／バドミントンサークルの運営アプリ）
- 技術: Next.js（App Router）＋ TypeScript ＋ Firebase
- Firebaseプロジェクト名: `team-management-service`（本番。所有者は yuka.uesugi@all-in-one.jp）
- 似た名前の別アプリ `pkanri` が隣にある。間違えないよう注意。

## 確認・デプロイの標準手順
- ローカル確認: `cd /Users/uesugiyuka/Desktop/antigravty/bigbeans && npm run dev` → `http://localhost:3001`（3000では開かない）

## つまずきやすい運用ルール
- **Firebaseアカウントの食い違い**: ローカルのfirebase CLIは `yuka-uesugi@b-w-c.jp` で権限がない。本番プロジェクトへのルール反映は**必ずFirebaseコンソール**の「ルール」タブから行う。Firestore用とStorage用のルール画面を間違えないこと（URLで見分ける: `.../firestore/.../security/rules` と `.../storage/...rules`）。
- **会計の書込権限**: 「完全に管理者だけ」にはできない（参加費が予約確定時に自動記録されるため）。完全限定化にはCloud Functionsでの再設計が必要。
- **サポーター権限**: 代理編集の仕組みあり。Firestoreルールは対応済み。
- **予約解禁スケジュール**: 規約と実装場所はプロジェクトメモリー参照。
- **申請の「承認」ボタンは署名の締め切りボタンでもある**（2026-07-20に実際の事故で判明）。
  種別変更の申請は署名が60%集まっても自動承認しない作りだが、承認すると status が voting → approved に変わり、
  署名ボタンの表示条件（[applications/page.tsx:522](src/app/dashboard/applications/page.tsx:522)）から外れて**未署名の人は二度と署名できなくなる**。
  「60%に達したから締め切られた」のではない。承認前に未署名人数を警告する表示を追加済み。
- **画面とFirestoreルールの食い違い**: 規約ページは `hasEditPermission = true` 固定で誰にでも編集ボタンが出る。
  保存先のルールが管理者限定だと「必ず失敗する」状態になる（当番表・SNS担当メモで実際に発生）。
  設定系の保存を追加するときは、ルール側の権限とセットで確認すること。
- **モーダルは必ず `createPortal` で body 直下に出す**: カード側に `animate-fade-in-up`（transform が残る）と
  `overflow-hidden` があると、中の `position: fixed` が画面基準でなくカード基準になり、スマホで画面外に出て
  白紙に見える。ビジター登録モーダルで実際に発生（2026-07-20修正）。
- **参加登録には「予約」と「出欠」の2つの入口があり、片方だけ直しても動かない**（2026-07-20に判明）。
  `joinPractice` は「予約(reservations)を作る → 確定したら出欠(attendances)に載せる」順で動く。
  2026-07-11に「ビジターのログイン不要参加」を許可したとき、**開放したのは attendances だけ**で
  reservations は `isMember()` のまま残っていた。予約ルール(bookingConfig)のある練習では
  最初の予約づくりで必ず失敗するため、**ビジター本人の予約は一度も成立していなかった**
  （本番の予約データを数えたら visitor が0件で発覚）。
  → 権限まわりを直したら、**本番データを実際に数えて裏を取る**こと。画面が出るだけでは動作確認にならない。
  → Firestoreの読み取りだけなら `curl` でREST APIを叩いて確認できる（events/reservations は誰でも読める）。
    例: `https://firestore.googleapis.com/v1/projects/team-management-service/databases/(default)/documents/events`
- **本番HPの確認に `curl` だけを使わない**: ヒーロー部分は `useSearchParams` を使うクライアント側の描画で、
  HTMLには含まれない。curlで「文言が無い」と出ても消えたとは限らない。必ずブラウザで開いて確かめる。
- **キャンセルは他人のデータを触る**: `cancelParticipation` は空き枠が出るとキャンセル待ちの先頭を
  自動で繰り上げる（＝他人の予約を update する）。権限を緩める前に必ずこの点を確認すること。
- 引き継ぎ再開の合言葉は「ビックビーンズ」（言われたら未了タスクから再開する）。
